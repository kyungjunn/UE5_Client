# 작업 내역 — UE5 계정 서버 통신 클라이언트

작성일: 2026-07-15

## 개요

언리얼 엔진 5.7 클라이언트(`UEClient_20260715`)에서 계정 백엔드 서버(FastAPI + MySQL)와
REST(JSON)로 통신하는 **C++ 클라이언트**를 구현했다. 회원가입 · 로그인 · 내 정보 조회 ·
회원탈퇴 4개 기능을 제공한다.

기존 프로젝트는 블루프린트 전용(Source 없음)이었으며, 이번 작업으로 **C++ 프로젝트로 전환**했다.

## 확정된 설계 결정

| 항목 | 결정 |
|------|------|
| 구현 방식 | C++ (언리얼 HTTP 모듈) |
| 아키텍처 | `UGameInstanceSubsystem` 단일 클래스 (A안) |
| 호출 방식 | `BlueprintCallable` 함수 + 콘솔 명령어 (둘 다) |
| 토큰 보관 | JWT 메모리 보관 (재시작 시 소멸) |
| 서버 주소 | 기본 `http://192.168.0.106:8088` — Ubuntu 서버, 5차에서 변경 (`SetBaseUrl`로 변경 가능) |

## 서버 계약 (서버 소스에서 직접 확인)

| API | 요청 본문 | 성공 응답 | 실패 |
|-----|-----------|-----------|------|
| POST `/auth/register` | `{email, nickname(2~50), password(8~128)}` | 201 `{id, email, nickname, created_at}` | 409 / 422 `{detail}` |
| POST `/auth/login` | `{email, password}` | 200 `{access_token, token_type}` | 401 `{detail}` |
| GET `/auth/me` | (Bearer) | 200 `{id, email, nickname, created_at}` | 401 |
| DELETE `/auth/withdraw` | (Bearer) | 204 (본문 없음) | 401 |

- 인증: `Authorization: Bearer <access_token>` (JWT, HS256)
- 에러 본문: FastAPI 표준 `{"detail": "..."}` (문자열) / 422는 `{"detail": [{"msg": ...}]}` (배열)

## 구현 결과

### 데이터 구조
- `FAccountUser` (USTRUCT, BlueprintType) — `Id`, `Email`, `Nickname`, `CreatedAt`

### 델리게이트 (다이내믹 멀티캐스트, BP 바인딩용)
- `FAccountSimpleResult(bool, int32, FString)` — 로그인 / 탈퇴
- `FAccountUserResult(bool, int32, FString, FAccountUser)` — 회원가입 / 내 정보

### UAccountSubsystem (UGameInstanceSubsystem)
- 상태: `BaseUrl`, `AuthToken`(메모리)
- `UFUNCTION(BlueprintCallable)`: `Register` / `Login` / `GetMe` / `Withdraw` / `SetBaseUrl`
- `BlueprintAssignable` 델리게이트 4개: `OnRegisterCompleted` / `OnLoginCompleted` / `OnMeCompleted` / `OnWithdrawCompleted`
- Me/Withdraw는 토큰 없으면 요청 없이 즉시 실패 처리

### 콘솔 명령어 (`Initialize`에서 `IConsoleManager`로 등록)
- `Account.Register <email> <nickname> <password>`
- `Account.Login <email> <password>`
- `Account.Me`
- `Account.Withdraw`
- `Account.SetBaseUrl <url>`
- 결과는 델리게이트 구독으로 `UE_LOG` 자동 출력

### 에러 처리
- 연결 실패 → `bSuccess=false, StatusCode=0, "서버에 연결할 수 없습니다."`
- 2xx → 성공 / 비2xx → `detail` 파싱 (문자열 그대로 / 422 배열은 첫 `msg`)
- 인증 필요 API인데 토큰 없음 → 요청 없이 `"로그인이 필요합니다."`
- 204(withdraw)는 본문 파싱 없이 성공 처리

## 2차 작업: 로그인 / 회원가입 UserWidget 연동 (C++ 베이스)

`UAccountSubsystem`을 호출하고 결과를 UI에 반영하는 UserWidget C++ 베이스 클래스를 추가했다.

### 구조
- `UAuthUserWidgetBase (UUserWidget, Abstract)` — 공통 베이스
  - `StatusText` (BindWidgetOptional) 상태 메시지 출력
  - `GetAccountSubsystem()`, `SetStatus(Message, bIsError)` (성공 초록 / 실패 빨강)
- `ULoginUserWidget` — BindWidget: `EmailInput`, `PasswordInput`, `LoginButton`, `SignupButton`
  - `RegisterWidgetClass` (TSubclassOf, EditAnywhere) — 띄울 회원가입 위젯 지정
  - BlueprintImplementableEvent: `OnLoginFinished(bSuccess, Message)`
- `URegisterUserWidget` — BindWidget: `EmailInput`, `NicknameInput`, `PasswordInput`, `RegisterButton`, `ExitButton`
  - BlueprintImplementableEvent: `OnRegisterFinished(bSuccess, Message)`

### 동작
- 버튼 클릭 → 입력값 공백 검사 → 버튼 비활성화 + "처리 중..." → 서브시스템 호출
- 완료 델리게이트 수신 → 버튼 재활성화 → `StatusText` 반영 → `On...Finished` BP 이벤트 호출
- `NativeConstruct`에서 델리게이트 구독, `NativeDestruct`에서 해제
- Build.cs 의존성에 `UMG`, `Slate`, `SlateCore` 추가

### 화면 전환 (Signup / Exit 버튼)
- 로그인 화면의 `SignupButton` → `RegisterWidgetClass`로 지정된 위젯을 생성해 `AddToViewport`(오버레이)
- 회원가입 화면의 `ExitButton` → `RemoveFromParent()`로 자신 제거 → 밑에 깔린 Login 화면 복귀
- 즉, Signup 클릭 → WBP_Register 오버레이 → X 클릭 → Login 복귀

### 블루프린트 연동
1. `LoginUserWidget` / `RegisterUserWidget`을 부모로 하는 Widget Blueprint 생성
2. 디자이너에서 위 BindWidget 이름과 **정확히 일치**하는 자식 위젯 배치
   - Login: `SignupButton` 버튼 추가 / Register: `ExitButton` 버튼 추가
3. `WBP_Login`의 Class Defaults → Account 카테고리 → `Register Widget Class`에 `WBP_Register` 지정
4. 성공 후 화면 전환 등은 `On Login Finished` / `On Register Finished` 이벤트에 연결

## 3차 작업: Lobby / 세션(방) 시스템

로그인 성공 후 **Lobby 레벨**로 이동해 프로필(닉네임)을 보여주고, 언리얼
**OnlineSubsystem(NULL/LAN)** 세션으로 **방 생성 / 방 참가**를 구현했다.

### 확정 설계
| 항목 | 결정 |
|------|------|
| OnlineSubsystem | NULL (LAN) — 리슨 서버 모델 |
| 레벨 구조 | Lobby(브라우저) 맵 + Room(실제 방) 맵 분리 |
| 프로필 표시 | 닉네임만 (로그인 후 `GetMe`로 취득) |
| 닉네임 보관 | 커스텀 `UGameInstance`에 캐시 |

### 레벨 흐름
```
[Login] --로그인 성공(GetMe)--> [Lobby (standalone): 프로필 + 방만들기 + RoomList]
  방 만들기 → CreateSession(max 4) → ServerTravel("Room?listen")
  방 참가   → JoinSession → ClientTravel → [Room (listen server)]
[Room] LobbyGameMode/PC/PlayerState → 접속자 닉네임 목록 / 나가기 → Lobby 복귀
```

### 추가/수정 클래스
| 클래스 | 역할 |
|--------|------|
| `UUEClientGameInstance` | 로그인 닉네임 캐시 (레벨 이동해도 유지) |
| `UGameSessionSubsystem` | 세션 래핑: `HostRoom`/`FindRooms`/`JoinRoomById`/`LeaveRoom`, `FRoomInfo`, 완료 델리게이트 |
| `ALobbyGameMode` | Room 맵 게임모드 (폰 없음, PC/PS 지정) |
| `ALobbyPlayerController` | 로컬 닉네임을 `ServerSetNickname` RPC로 서버 전달 |
| `ALobbyPlayerState` | 복제되는 `Nickname` |
| `ULobbyWidget` | 프로필 + 방만들기 + RoomList(병합) |
| `URoomListEntryWidget` | 방 한 줄 (꽉 차면 Join 비활성) |
| `URoomWidget` | 방 안 닉네임 목록 + 나가기 |
| `ULoginUserWidget` (수정) | 로그인 성공 → `GetMe` → 닉네임 캐시 → `OpenLevel(Lobby)` |

### 주요 동작
- 방 세션: `NumPublicConnections=4`, `bIsLANMatch=true`, 커스텀 설정 `ROOMNAME`/`HOSTNAME` 저장
- RoomList: `RoomId` 기준 **병합** — 새 방 추가·기존 방 갱신·**검색에서 빠진 방은 제거하지 않음** *(→ 5차에서 제거 방식으로 변경)*
- 꽉 찬 방(`CurrentPlayers ≥ MaxPlayers`)은 Join 버튼 비활성
- 방 안 닉네임: 클라 `BeginPlay`→`ServerSetNickname` RPC→`PlayerState` 복제, 위젯은 `GameState->PlayerArray`를 0.5초 타이머로 갱신

### 설정 변경
- `Build.cs`: `OnlineSubsystem`, `OnlineSubsystemUtils` 추가
- `.uproject`: `OnlineSubsystemNull`, `OnlineSubsystemUtils` 플러그인 활성화
- `DefaultEngine.ini`: `GameInstanceClass` 지정, `[OnlineSubsystem] DefaultPlatformService=Null`

### 에디터 연동 (BindWidget 이름)
| WBP | 부모 | 배치할 위젯 |
|-----|------|-------------|
| `WBP_Lobby` | `LobbyWidget` | `NicknameText`, `RoomNameInput`, `CreateRoomButton`, `RefreshButton`, `RoomListBox` |
| `WBP_RoomEntry` | `RoomListEntryWidget` | `RoomNameText`, `HostNameText`, `CountText`, `JoinButton` |
| `WBP_Room` | `RoomWidget` | `PlayerListBox`, `LeaveButton` |

- `WBP_Lobby` → Class Defaults → Session → `Room Entry Widget Class` = `WBP_RoomEntry`
- `Room` 맵 → World Settings → `GameMode Override` = `LobbyGameMode`
- Lobby/Room 맵 Level Blueprint(BeginPlay)에서 각 위젯 Create + Add to Viewport

## 4차 작업: Lobby/Room UI 에셋 파이프라인 및 PIE 표시 버그 수정

Lobby/Room 화면에 필요한 블루프린트·위젯 에셋을 **C++ 에디터 도구로 자동 생성**하고,
위젯이 화면에 나오지 않던 문제들을 해결했다.

### 위젯 표시 방식 변경 (Level BP → PlayerController)
- `ALobbyPlayerController`에 `LobbyWidgetClass`(TSubclassOf\<UUserWidget\>, EditAnywhere) 추가
- `BeginPlay`(로컬 컨트롤러)에서 위젯 생성 + `AddToViewport` + 마우스 커서 표시 + **`FInputModeUIOnly`**
- 리슨서버 호스트/클라이언트 모두 각자의 로컬 컨트롤러에서 실행 → 3차의 "Level Blueprint에서 Create + Add to Viewport" 방식을 대체

### LobbyAssetGenerator (에디터 전용 C++ 도구)
- UE 5.7 Python에서는 `WidgetBlueprint.WidgetTree`/`RootWidget`이 protected라 순수 Python으로 위젯 트리 구성 불가
  → `ULobbyAssetGenerator`(BlueprintFunctionLibrary, `WITH_EDITOR`)를 만들어 **Python 커맨드릿에서 호출**
- 기존 에셋이 있으면 위젯 트리를 **in-place 재구성** (UClass 유지 → CDO/맵의 외부 참조 보존), 멱등 실행 가능
- `Build.cs`: `bBuildEditor`일 때 `UnrealEd`/`UMGEditor`/`AssetTools` 추가, `.uproject`에 `PythonScriptPlugin` 활성화

### 생성된 에셋
| 에셋 | 내용 |
|------|------|
| `Content/Lobby/Widget/WBP_Lobby` | 로비 UI — 제목/닉네임/방 이름 입력/방 만들기/새로고침/방 리스트(ScrollBox) |
| `Content/Lobby/Widget/WBP_RoomEntry` | 방 리스트 한 줄 — **Border 루트** (방 이름/호스트/인원/참가) |
| `Content/Lobby/Widget/WBP_Room` | 방 UI — 제목/참가자 목록/나가기 |
| `Content/Lobby/BP_LobbyPC` / `BP_RoomPC` | `LobbyPlayerController` 상속, `LobbyWidgetClass`=WBP_Lobby / WBP_Room |
| `Content/Lobby/BP_RoomGM` | `LobbyGameMode` 상속, PC=BP_RoomPC |
| `Content/Maps/Room.umap` | Lobby 복제 + World Settings GameMode Override=BP_RoomGM |

- 전 위젯 1920×1080 기준 **풀스크린 앵커(0,0→1,1) 레이아웃** (해상도에 맞춰 스트레치)
- Lobby 맵은 GameMode Override=`BP_LobbyGM`(PC=BP_LobbyPC, 사용자 구성)

### 해결한 버그 3건
1. **WBP_Lobby를 바인딩해도 화면에 안 보임** — 위젯 내용이 비어 있었음(콘텐츠/크기 0) → 전체 UI 레이아웃 구성으로 해결
2. **방 만들기 → 로그인 화면 등장** — `/Game/Maps/Room` 맵이 없어 `ServerTravel` 실패 → `GameDefaultMap`(Login)으로 폴백된 것. Room 맵 + BP_RoomGM 생성으로 해결
3. **방 리스트 항목이 안 보임** — `WBP_RoomEntry` 루트가 CanvasPanel이라 ScrollBox 안에서 desired size 0 → **Border 루트**로 교체해 해결

### 검증
- `-game -windowed` 실행 스크린샷으로 Lobby UI / Room UI / 방 리스트 렌더링 확인

## 5차 작업: Ubuntu 실서버 연동 + 멀티 인스턴스 안정화

### 서버 주소 변경
- 계정 서버(FastAPI + MySQL)를 **Ubuntu 머신으로 이전** → `BaseUrl` 기본값을 `http://192.168.0.106:8088`로 변경

### 실서버 엔드투엔드 검증 (실제 게임 클라이언트 실행)
```
[Account] 성공 (HTTP 201) 회원가입 성공 | id=2 email=claudetest01@test.com nick=클로드테스터
[Account] 성공 (HTTP 200) 로그인 성공
[Account] 성공 (HTTP 200) 내 정보 조회 성공 | id=2 ...
```
- 한글 닉네임 UTF-8 저장/조회 정상, JWT Bearer 토큰 발급→저장→`/auth/me` 인증 체인 확인
- 최초 500 오류의 원인은 서버 DB `users` 테이블에 `deleted_at` 컬럼 누락 →
  `ALTER TABLE users ADD COLUMN deleted_at DATETIME NULL` (서버 측 조치)로 해결

### 버그 수정 3건
1. **PIE 2창(리슨서버) 종료 시 크래시** — GameInstance 2개가 같은 이름의 콘솔 명령(`Account.*`)을 중복 등록
   → 콘솔 매니저가 같은 오브젝트를 공유시켜 종료 시 **이중 해제**(access violation).
   `FindConsoleObject` 가드로 첫 인스턴스만 등록하도록 수정 (`AccountSubsystem.cpp`)
2. **다른 클라이언트에서 새로고침해도 방 리스트 안 뜸** — `IOnlineSubsystem::Get()`이 프로세스 전역 인스턴스를
   반환해 PIE 다중창이 OSS를 공유 → LAN 비콘 광고/검색 충돌(`Ignoring game search request while one is pending`).
   `Online::GetSubsystem(GetWorld())`로 **인스턴스별 OSS** 사용 + 검색 진행 중(`InProgress`) 재요청 가드 추가 (`GameSessionSubsystem.cpp`)
3. **사라진 방이 리스트에 계속 남음** — ①위 전역 OSS 공유 때문에 PIE 종료 후에도 에디터 프로세스에 유령 세션이
   남아 광고됐던 문제는 2번 수정으로 해소(이제 PIE 종료 시 세션도 소멸),
   ②위젯 병합 로직을 **동기화 방식으로 변경**: 최신 검색 결과에 없는 방은 리스트에서 제거 (`LobbyWidget.cpp`)

### 검증
- 빌드 성공 + 게임 실행 콘솔 명령 회귀 테스트(로그인/내 정보 정상)
- PIE 리슨서버 2창에서 방 생성 → 새로고침 → 리스트 표시 → 참가 → 종료(크래시 없음) 정상 동작 확인

## 6차 작업: NPC 채팅 시스템 (로컬 LLM: llama.cpp + Gemma)

로비 레벨에서 플레이어가 NPC "세라"와 채팅하는 시스템. LLM 백엔드는 로컬
llama.cpp 서버(`http://127.0.0.1:8080`, 모델 `gemma-4-E2B-it-Q8_0`)의
**OpenAI 호환 `/v1/chat/completions`** 를 사용한다.

### 확정 설계
| 항목 | 결정 |
|------|------|
| NPC | 기본 1명 — "세라" (밝고 친절한 로비 안내원) |
| 초상화 | 아트 에셋 없음 → 성격 색상 Border + 이니셜 플레이스홀더 |
| 대화 기억 | 단발 질문-답변 (히스토리 미포함) |
| 배치 | 로비 우측 세로 패널 (우상단 초상화+이름 / 대화 로그 / 하단 입력+전송) |

### UNpcChatSubsystem (UGameInstanceSubsystem, 신규)
- **`FNpcProfile` 구조체**: `Name` / `Personality`(성격 지침) / `StatusLine`(상태 정보) / `PortraitColor`
- **요청 페이로드 조립** (`BuildRequestBody`): system 메시지 = `"당신은 '{이름}'입니다. {성격}\n현재 상태: {상태}\n답변은 한국어로 2~3문장 이내로 짧게 하세요."` + user 메시지, `max_tokens 1024` / `temperature 0.7`
- **비동기 + 델리게이트**: HTTP 완료 시 `OnChatResponseReceived(bool, NpcName, Message)` (BlueprintAssignable) 브로드캐스트. 연결 실패/비2xx/빈 응답은 bSuccess=false + 에러 설명
- 진행 중 요청 가드(`bRequestInFlight`), `SetLlmBaseUrl`
- 콘솔 명령: `Npc.Chat <message>` / `Npc.SetLlmUrl <url>` (중복 등록 가드 포함)

### UNpcChatWidget (신규) + 에셋
- BindWidget: `PortraitBorder`/`NpcInitialText`(초상화), `NpcNameText`, `ChatLogBox`(ScrollBox), `ChatInput`, `SendButton`
- 전송(버튼/Enter) → "나: ..." 추가 + 버튼 비활성 + "세라: ..." 대기 표시 → 응답으로 교체 + 자동 스크롤
- `NpcProfile` 기본값은 세라 (BP Class Defaults 에서 이름/성격/상태/색 변경 가능)
- `ULobbyWidget`: 우측 `ChatPanelSlot`(BindWidgetOptional) + `NpcChatWidgetClass` → NativeConstruct 에서 채팅 위젯 삽입
- `LobbyAssetGenerator::GenerateNpcChatAssets()`: **WBP_NpcChat** 생성 + **WBP_Lobby 2컬럼 재구성**(왼쪽 로비 fill + 오른쪽 채팅 고정폭 420px) + CDO 배선

## 7차 작업: InGame 레벨 + 준비/시작 플로우 + 필드 NPC 대화

방에서 준비/시작을 거쳐 전원이 **InGame 레벨**로 이동, ThirdPerson 캐릭터로 돌아다니며
필드 NPC 에게 접근해 **F 키로 LLM 채팅**하는 기능. 사용자가 추가한 에셋
(CityFolks 캐릭터 팩, ThirdPerson 템플릿, EnhancedInput Content/Input) 활용.

### 확정 설계
| 항목 | 결정 |
|------|------|
| NPC | 2명 — 도우 센세이(요리 사부) + 데포르티스타(운동선수), 인스턴스별 프로필 |
| 시작 조건 | 호스트 제외 전원 준비 시 시작 가능 (혼자면 즉시 가능) |
| 채팅 닫기 | 입력창 포커스 없을 때 F + ESC 항상 (한글 ㄹ=F키 오작동 방지) |
| InGame 맵 | Lvl_ThirdPerson 복제(`Maps/InGame/InGame.umap`) + 바닥 ×4 + NPC 배치 |

### 준비/시작 플로우 (Room)
- `ALobbyPlayerState.bIsReady` 복제 추가, `ALobbyPlayerController` 에 `ServerSetReady` / `ServerRequestStartGame` RPC
- `URoomWidget`: 클라 전용 **준비** 토글 / 호스트 전용 **시작** 버튼(전원 준비 시 활성), 목록에 `[준비완료]` 표시
- 시작 → `ServerTravel("/Game/Maps/InGame/InGame?listen")` 로 방 전원 동반 이동

### InGame 구성 (신규 C++ 4종 + 에셋)
| 클래스/에셋 | 역할 |
|------|------|
| `AInGameGameMode` + `BP_InGameGM` | Pawn=BP_ThirdPersonCharacter, PC=BP_InGamePC, PlayerState=LobbyPlayerState(닉네임 유지) |
| `AInGamePlayerController` + `BP_InGamePC` | F(IA_Interact) 바인딩, 프롬프트/채팅 토글, 채팅 중 IMC 제거로 이동·점프·시점 차단 |
| `ANpcActor` | 스켈레탈 메시 + 반경 250 거리 체크(0.25s) → 프롬프트 표시. `FNpcProfile`/IdleAnim 인스턴스별 설정 |
| `UInGameChatWidget` + `WBP_InGameChat` | 하단 도킹 검정 반투명(0.65) 채팅 바 — ESC 로 닫기 요청, NPC 프로필 주입 |
| `WBP_InteractPrompt` | "[F] 대화 시작" 하단 중앙 프롬프트 |
| `IA_Interact` / `IMC_Interact` | EnhancedInput F 키 (Content/Input, 기존 IMC_Default 는 무변경) |

- LLM 통신은 6차의 `NpcChatSubsystem` 그대로 재사용 (NPC별 성격 프롬프트만 교체)
- `Build.cs`: `EnhancedInput`, `InputCore` 추가

### 해결한 이슈 4건
1. **BP CDO 배선이 디스크에 저장 안 됨** — `save_dirty_packages` 로는 리플렉션 CDO 수정이 저장되지 않음
   → `save_directory(only_if_is_dirty=False)` 강제 저장 + 별도 프로세스로 디스크 재검증
2. **`MapKey()` 가 저장에 반영 안 됨** — C++/Python 모두 호출 직후에도 mappings 0개
   → Python 에서 `mappings` 프로퍼티를 직접 설정
3. **프롬프트가 안 뜸** — 폰 스폰 시 초기 오버랩이 빙의(possess) 전에 발생해 이벤트 유실
   → 오버랩 대신 저빈도(0.25초) 거리 체크로 교체
4. **채팅 패널 높이 40px** — 비스트레치 Y 앵커에서는 `Offsets.Bottom` 이 여백이 아니라 **높이** → 320 으로 수정
