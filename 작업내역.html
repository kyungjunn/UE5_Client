작업 내역 — UE5 계정 서버 통신 클라이언트

작성일: 2026-07-15

개요

언리얼 엔진 5.7 클라이언트(UEClient_20260715)에서 계정 백엔드 서버(FastAPI + MySQL)와 REST(JSON)로 통신하는 C++ 클라이언트를 구현했다. 회원가입 · 로그인 · 내 정보 조회 · 회원탈퇴 4개 기능을 제공한다.

기존 프로젝트는 블루프린트 전용(Source 없음)이었으며, 이번 작업으로 C++ 프로젝트로 전환했다.

확정된 설계 결정

항목결정
구현 방식C++ (언리얼 HTTP 모듈)
아키텍처UGameInstanceSubsystem 단일 클래스 (A안)
호출 방식BlueprintCallable 함수 + 콘솔 명령어 (둘 다)
토큰 보관JWT 메모리 보관 (재시작 시 소멸)
서버 주소기본 http://192.168.0.106:8088 — Ubuntu 서버, 5차에서 변경 (SetBaseUrl로 변경 가능)

서버 계약 서버 소스에서 직접 확인

API요청 본문성공 응답실패
POST /auth/registeremail, nickname(2~50), password(8~128)201 (id, email, nickname, created_at)409 / 422 detail
POST /auth/loginemail, password200 (access_token, token_type)401 detail
GET /auth/me(Bearer)200 (id, email, nickname, created_at)401
DELETE /auth/withdraw(Bearer)204 (본문 없음)401
  • 인증: Authorization: Bearer <access_token> (JWT, HS256)
  • 에러 본문: FastAPI 표준 {"detail": "..."} (문자열) / 422는 {"detail": [{"msg": ...}]} (배열)

구현 결과

데이터 구조 / 델리게이트

  • FAccountUser (USTRUCT, BlueprintType) — Id, Email, Nickname, CreatedAt
  • FAccountSimpleResult(bool, int32, FString) — 로그인 / 탈퇴
  • FAccountUserResult(bool, int32, FString, FAccountUser) — 회원가입 / 내 정보

UAccountSubsystem (UGameInstanceSubsystem)

  • 상태: BaseUrl, AuthToken(메모리)
  • UFUNCTION(BlueprintCallable): Register / Login / GetMe / Withdraw / SetBaseUrl
  • BlueprintAssignable 델리게이트 4개: OnRegisterCompleted / OnLoginCompleted / OnMeCompleted / OnWithdrawCompleted
  • Me/Withdraw는 토큰 없으면 요청 없이 즉시 실패 처리

콘솔 명령어

Account.Register <email> <nickname> <password>
Account.Login <email> <password>
Account.Me
Account.Withdraw
Account.SetBaseUrl <url>

결과는 델리게이트 구독으로 UE_LOG 자동 출력 (콘솔·BP 어디서 호출하든).

에러 처리

  • 연결 실패 → bSuccess=false, StatusCode=0, "서버에 연결할 수 없습니다."
  • 2xx → 성공 / 비2xx → detail 파싱 (문자열 그대로 / 422 배열은 첫 msg)
  • 인증 필요 API인데 토큰 없음 → 요청 없이 "로그인이 필요합니다."
  • 204(withdraw)는 본문 파싱 없이 성공 처리

2차 작업: 로그인 / 회원가입 UserWidget 연동 (C++ 베이스)

UAccountSubsystem을 호출하고 결과를 UI에 반영하는 UserWidget C++ 베이스 클래스를 추가했다.

  • UAuthUserWidgetBase (UUserWidget, Abstract) — 공통 베이스
    • StatusText (BindWidgetOptional) 상태 메시지 출력
    • GetAccountSubsystem(), SetStatus(Message, bIsError) (성공 초록 / 실패 빨강)
  • ULoginUserWidget — BindWidget: EmailInput, PasswordInput, LoginButton, SignupButton / RegisterWidgetClass(TSubclassOf) / BlueprintImplementableEvent OnLoginFinished(bSuccess, Message)
  • URegisterUserWidget — BindWidget: EmailInput, NicknameInput, PasswordInput, RegisterButton, ExitButton / BlueprintImplementableEvent OnRegisterFinished(bSuccess, Message)

화면 전환: 로그인의 SignupButton → RegisterWidgetClass 위젯 생성 후 AddToViewport(오버레이); 회원가입의 ExitButton → RemoveFromParent()로 자신 제거 → 밑의 Login 복귀. (Signup 클릭 → WBP_Register 오버레이 → X 클릭 → Login 복귀)

동작: 버튼 클릭 → 입력값 공백 검사 → 버튼 비활성화 + "처리 중..." → 서브시스템 호출 → 완료 델리게이트 수신 → 버튼 재활성화 → StatusText 반영 → On...Finished BP 이벤트 호출. 구독은 NativeConstruct, 해제는 NativeDestruct. Build.cs에 UMG·Slate·SlateCore 추가.

3차 작업: Lobby / 세션(방) 시스템

로그인 성공 후 Lobby 레벨로 이동해 프로필(닉네임)을 보여주고, 언리얼 OnlineSubsystem(NULL/LAN) 세션으로 방 생성 / 방 참가를 구현했다.

확정 설계

항목결정
OnlineSubsystemNULL (LAN) — 리슨 서버 모델
레벨 구조Lobby(브라우저) 맵 + Room(실제 방) 맵 분리
프로필 표시닉네임만 (로그인 후 GetMe로 취득)
닉네임 보관커스텀 UGameInstance에 캐시

레벨 흐름

[Login] --로그인 성공(GetMe)--> [Lobby (standalone): 프로필 + 방만들기 + RoomList]
  방 만들기 → CreateSession(max 4) → ServerTravel("Room?listen")
  방 참가   → JoinSession → ClientTravel → [Room (listen server)]
[Room] LobbyGameMode/PC/PlayerState → 접속자 닉네임 목록 / 나가기 → Lobby 복귀

추가/수정 클래스

URoomListEntryWidget방 한 줄 (꽉 차면 Join 비활성)
URoomWidget방 안 닉네임 목록 + 나가기
ULoginUserWidget (수정)로그인 성공 → GetMe → 닉네임 캐시 → OpenLevel(Lobby)

주요 동작

  • 방 세션: NumPublicConnections=4, bIsLANMatch=true, 커스텀 설정 ROOMNAME/HOSTNAME 저장
  • RoomList: RoomId 기준 병합 — 새 방 추가·기존 방 갱신·검색에서 빠진 방은 제거하지 않음 (→ 5차에서 제거 방식으로 변경)
  • 꽉 찬 방(CurrentPlayers ≥ MaxPlayers)은 Join 버튼 비활성
  • 방 안 닉네임: 클라 BeginPlay→ServerSetNickname RPC→PlayerState 복제, 위젯은 GameState->PlayerArray를 0.5초 타이머로 갱신

설정 변경

  • Build.cs: OnlineSubsystem, OnlineSubsystemUtils 추가
  • .uproject: OnlineSubsystemNull, OnlineSubsystemUtils 플러그인 활성화
  • DefaultEngine.ini: GameInstanceClass 지정, [OnlineSubsystem] DefaultPlatformService=Null

4차 작업: Lobby/Room UI 에셋 파이프라인 및 PIE 표시 버그 수정

Lobby/Room 화면에 필요한 블루프린트·위젯 에셋을 C++ 에디터 도구로 자동 생성하고, 위젯이 화면에 나오지 않던 문제들을 해결했다.

위젯 표시 방식 변경 (Level BP → PlayerController)

  • ALobbyPlayerController에 LobbyWidgetClass(TSubclassOf<UUserWidget>, EditAnywhere) 추가
  • BeginPlay(로컬 컨트롤러)에서 위젯 생성 + AddToViewport + 마우스 커서 표시 + FInputModeUIOnly
  • 리슨서버 호스트/클라이언트 모두 각자의 로컬 컨트롤러에서 실행 → 3차의 "Level Blueprint에서 Create + Add to Viewport" 방식을 대체

LobbyAssetGenerator (에디터 전용 C++ 도구)

  • UE 5.7 Python에서는 WidgetBlueprint.WidgetTree/RootWidget이 protected라 순수 Python으로 위젯 트리 구성 불가 → ULobbyAssetGenerator(BlueprintFunctionLibrary, WITH_EDITOR)를 만들어 Python 커맨드릿에서 호출
  • 기존 에셋이 있으면 위젯 트리를 in-place 재구성 (UClass 유지 → CDO/맵의 외부 참조 보존), 멱등 실행 가능
  • Build.cs: bBuildEditor일 때 UnrealEd/UMGEditor/AssetTools 추가, .uproject에 PythonScriptPlugin 활성화

Content/Lobby/Widget/WBP_RoomEntry방 리스트 한 줄 — Border 루트 (방 이름/호스트/인원/참가)

  • 전 위젯 1920×1080 기준 풀스크린 앵커(0,0→1,1) 레이아웃 (해상도에 맞춰 스트레치)
  • Lobby 맵은 GameMode Override=BP_LobbyGM(PC=BP_LobbyPC, 사용자 구성)

해결한 버그 3건

  1. WBP_Lobby를 바인딩해도 화면에 안 보임 — 위젯 내용이 비어 있었음(콘텐츠/크기 0) → 전체 UI 레이아웃 구성으로 해결
  2. 방 만들기 → 로그인 화면 등장 — /Game/Maps/Room 맵이 없어 ServerTravel 실패 → GameDefaultMap(Login)으로 폴백된 것. Room 맵 + BP_RoomGM 생성으로 해결
  3. 방 리스트 항목이 안 보임 — WBP_RoomEntry 루트가 CanvasPanel이라 ScrollBox 안에서 desired size 0 → Border 루트로 교체해 해결

검증: -game -windowed 실행 스크린샷으로 Lobby UI / Room UI / 방 리스트 렌더링 확인.

5차 작업: Ubuntu 실서버 연동 + 멀티 인스턴스 안정화

서버 주소 변경

계정 서버(FastAPI + MySQL)를 Ubuntu 머신으로 이전 → BaseUrl 기본값을 http://192.168.0.106:8088로 변경.

실서버 엔드투엔드 검증 (실제 게임 클라이언트 실행)

[Account] 성공 (HTTP 201) 회원가입 성공 | id=2 email=claudetest01@test.com nick=클로드테스터
[Account] 성공 (HTTP 200) 로그인 성공
[Account] 성공 (HTTP 200) 내 정보 조회 성공 | id=2 ...
  • 한글 닉네임 UTF-8 저장/조회 정상, JWT Bearer 토큰 발급→저장→/auth/me 인증 체인 확인
  • 최초 500 오류의 원인은 서버 DB users 테이블에 deleted_at 컬럼 누락 → ALTER TABLE users ADD COLUMN deleted_at DATETIME NULL (서버 측 조치)로 해결

버그 수정 3건

  1. PIE 2창(리슨서버) 종료 시 크래시 — GameInstance 2개가 같은 이름의 콘솔 명령(Account.*)을 중복 등록 → 콘솔 매니저가 같은 오브젝트를 공유시켜 종료 시 이중 해제(access violation). FindConsoleObject 가드로 첫 인스턴스만 등록하도록 수정 (AccountSubsystem.cpp)
  2. 다른 클라이언트에서 새로고침해도 방 리스트 안 뜸 — IOnlineSubsystem::Get()이 프로세스 전역 인스턴스를 반환해 PIE 다중창이 OSS를 공유 → LAN 비콘 광고/검색 충돌(Ignoring game search request while one is pending). Online::GetSubsystem(GetWorld())로 인스턴스별 OSS 사용 + 검색 진행 중(InProgress) 재요청 가드 추가 (GameSessionSubsystem.cpp)
  3. 사라진 방이 리스트에 계속 남음 — ① 위 전역 OSS 공유 때문에 PIE 종료 후에도 에디터 프로세스에 유령 세션이 남아 광고됐던 문제는 2번 수정으로 해소(이제 PIE 종료 시 세션도 소멸), ② 위젯 병합 로직을 동기화 방식으로 변경: 최신 검색 결과에 없는 방은 리스트에서 제거 (LobbyWidget.cpp)

검증: 빌드 성공 + 게임 실행 콘솔 명령 회귀 테스트(로그인/내 정보 정상), PIE 리슨서버 2창에서 방 생성 → 새로고침 → 리스트 표시 → 참가 → 종료(크래시 없음) 정상 동작 확인.

검증 상태

  • ✅ 컴파일·링크 성공 — Build.bat UEClient_20260715Editor Win64 Development
    • 1차: 계정 통신 모듈 컴파일 → UnrealEditor-UEClient_20260715.dll 생성 확인

    • 2차: 로그인/가입 위젯 클래스 3종 컴파일·링크 성공

# 3) Play(PIE) 후 콘솔(백틱 ` 키)에서 순서대로:

Lobby / 세션 테스트

  1. 서버 기동 후, Editor Play 모드 = Standalone, Number of Players = 2 (PIE 다중창은 GameInstance 공유로 닉네임 이슈 → Standalone 권장)
  2. 각 인스턴스에서 서로 다른 계정으로 로그인 → 자동으로 Lobby 진입, 프로필에 닉네임 표시
  3. 한쪽에서 방 이름 입력 후 방 만들기 → Room 맵으로 이동(리슨 호스트)
  4. 다른쪽에서 새로고침 → RoomList에 방 표시(호스트/인원) → 참가 → Room에서 서로 닉네임 확인
  5. 인원 4/4인 방은 Join 버튼 비활성 확인, 새로고침 시 사라진 방이 목록에서 제거되는지 확인
